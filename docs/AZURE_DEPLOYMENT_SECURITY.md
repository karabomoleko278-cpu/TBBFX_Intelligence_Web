# TBBFX Azure Static Web Apps Deployment And Security Runbook

## Target Architecture

TBBFX should be deployed as a public read-only intelligence terminal first.

- Public frontend: Azure Static Web Apps Free tier hosts `terminal/`.
- Public analytics API: Azure Static Web Apps managed Functions in `api/`; currently only `/api/health` is enabled.
- Live push: Azure SignalR Service serverless negotiation is implemented at `/api/negotiate`.
- Private live ingestion: `/api/pulse` uses Function-level auth and broadcasts sanitized feature packets to Azure SignalR.
- Private execution: MT5 bridge, trade execution, validation optimizer, and `live_strategy_config.json` stay off the public web.
- Secrets: Azure Portal/GitHub secrets/Key Vault only. Do not commit `.env`, API keys, MT5 credentials, broker IDs, or SignalR connection strings.

## Public Endpoint Policy

Allowed:

- `/`
- `/terminal`
- `/orderflow`
- `/api/health`
- `/api/negotiate`
- `/config.public.js`
- Static JS/HTML/CSS/assets

Private/authenticated only:

- `/api/pulse`

Blocked in `staticwebapp.config.json`:

- `/api/validation/*`
- `/api/live-scanner/*`
- `/features/update`
- `/features/all`
- `/hub/*`

The blocked routes are intentionally strict. They prevent public users from triggering CPU-heavy backtests, reading local live strategy configuration, posting fake feature packets, or connecting to a private SignalR hub by accident.

## Existing Backend Hardening

`SignalRFeatureStore` now includes:

- Configurable CORS via `TBBFX_ALLOWED_ORIGINS`.
- Fixed-window rate limits for public reads and private writes.
- Browser security headers: `nosniff`, `DENY` framing, strict referrer policy, and locked-down permissions policy.
- Optional public read-only kill switch via `TBBFX_PUBLIC_READONLY=true`.
- Optional ingestion API key via `TBBFX_FEATURE_UPDATE_KEY`.
- Azure-compatible port selection via `PORT`.
- CI-safe project build target so `taskkill` does not run inside GitHub/Azure builds.

## Azure SignalR Serverless

The public terminal uses `config.public.js` with:

- `signalRUrl: "/api"`
- `signalRMode: "azure-signalr-serverless"`

The SignalR JavaScript client calls `/api/negotiate`; Azure Functions returns an Azure SignalR access URL/token using the `AzureSignalRConnectionString` app setting.

The browser is receive-only in this mode. It does not call `JoinSymbolGroup`, `LeaveSymbolGroup`, trade methods, or validation methods.

Your private live feeder can publish sanitized market feature packets to:

- `POST /api/pulse?code=<FUNCTION_KEY>`

Never put the function key in frontend JavaScript.

## Application Insights

Enable Application Insights on the Static Web App / managed Functions and set:

- `APPLICATIONINSIGHTS_CONNECTION_STRING`

Use it to monitor:

- failed negotiate attempts
- 429 rate-limit responses
- unexpected public access attempts to blocked routes
- function exceptions
- traffic spikes by IP/user agent

## API Management / Front Door WAF

For a serious public launch, place one of these in front of the Static Web App/API:

- API Management policy: `infra/apim-readonly-rate-limit-policy.xml`
- Front Door WAF template: `infra/front-door-waf-policy.template.json`

Do not deploy Front Door Premium/WAF without accepting the Azure cost. The template is included so the controls are ready, but it should be enabled deliberately.

## Required Azure/GitHub Secrets

GitHub repository secret:

- `AZURE_STATIC_WEB_APPS_API_TOKEN`: generated by Azure Static Web Apps during app creation.

If a separate backend is added later:

- `TBBFX_ALLOWED_ORIGINS`: semicolon-separated origins, for example `https://your-app.azurestaticapps.net;https://yourcustomdomain.com`.
- `TBBFX_PUBLIC_READONLY=true` for public read-only services.
- `TBBFX_FEATURE_UPDATE_KEY`: long random key required for feature ingestion.
- `AzureSignalRConnectionString`: store in Azure App Settings or Key Vault, never in source.
- `APPLICATIONINSIGHTS_CONNECTION_STRING`: store in Azure App Settings or Key Vault.
- Broker/MT5 credentials: never expose to Static Web Apps or browser JavaScript.

## Caching

- `config.public.js` is `no-store` so runtime public/private flags can be changed quickly.
- JavaScript files are cached for one hour.
- Market data APIs, once added, should use short TTLs:
  - live candle snapshots: 3-15 seconds
  - historical candles: 5-60 minutes
  - macro/event cache: 15-60 minutes

Do not cache authenticated/admin responses in the browser.

## Rate Limits

Current backend limits:

- Public reads: 120 requests per minute per limiter bucket.
- Private writes: 30 requests per minute.

For a real public launch with traffic, add one of:

- Azure API Management consumption tier in front of Functions.
- Azure Front Door with WAF/rate-limit rules.
- Cloudflare proxy/WAF if you choose a custom domain outside Azure.

## Deployment Steps

1. Push `TBBFX_Intelligence_Web` to GitHub.
2. In Azure Portal, create `Static Web App`.
3. Choose Free plan.
4. Connect GitHub repo.
5. Set:
   - App location: `terminal`
   - Api location: `api`
   - Output location: blank
6. Let Azure create or reuse `.github/workflows/azure-static-web-apps.yml`.
7. Add custom domain after the first successful deployment.
8. Replace `YOUR_PUBLIC_TBBFX_DOMAIN` in `terminal/robots.txt` and `terminal/sitemap.xml`.
9. Submit the sitemap in Bing Webmaster Tools and Google Search Console.
10. In Azure Static Web Apps > Configuration, add `AzureSignalRConnectionString` and `APPLICATIONINSIGHTS_CONNECTION_STRING`.
11. If serious public traffic starts, place API Management or Front Door WAF in front of the public API.

## What Must Not Be Public

- MT5 account login, password, server, investor password, broker account number.
- Any endpoint that can place, modify, or close trades.
- Local relay port `127.0.0.1:77`.
- `live_strategy_config.json`.
- Validation runner endpoints.
- Feature ingestion endpoints without authentication.
- SignalR service connection strings.

## Future Production Upgrade

When you want real live data publicly:

1. Put read-only market data into Azure Functions.
2. Put SignalR behind Azure SignalR Service.
3. Require API keys or managed identity for ingestion.
4. Keep execution as a private service on your machine/VPS.
5. Use Application Insights for error telemetry and suspicious traffic visibility.
6. Add WAF/rate limiting before heavy usage.

This keeps the public terminal sellable while avoiding a direct line from anonymous internet users to your EA or broker account.
